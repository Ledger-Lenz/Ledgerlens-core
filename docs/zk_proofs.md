# ZK Threshold Proof Protocol Specification

This document defines the Fiat-Shamir transcript format and verification logic
for the LedgerLens ZK threshold proof scheme. The protocol proves that a
Pedersen-committed score `s` satisfies `s >= t` for a public threshold `t`
without revealing `s`.

## Cryptographic Primitives

- **Curve**: BN254 (alt_bn128) — the pairing-friendly elliptic curve used by
  Ethereum precompiles and Soroban.
- **Curve arithmetic**: `py_ecc.bn128` (Python), `ark-bn254` (Rust).
- **Hash function**: SHA-256.

## Pedersen Commitment

A score `s` (0–100) is committed as:

```
C = s * G + r * H
```

where:
- `G` is the BN254 generator point `G1`.
- `H` is a second generator derived as `H = SHA256("LedgerLens ZK Generator H") * G1`,
  ensuring the discrete-log relation between `G` and `H` is unknown.
- `r` is a random blinding factor (prover secret).

## Bit Decomposition

The value `d = s - t` (score minus threshold) is decomposed into `NUM_BITS = 7`
bits (2^7 = 128 >= 100). Each bit `b_i ∈ {0, 1}` is committed as:

```
B_i = b_i * G + r_i * H
```

The overall blinding factor is `r = Σ (2^i * r_i) mod q`.

## Fiat-Shamir Transcript

The non-interactive challenge is computed as:

```
context = SHA256(wallet || t_byte || C.x_bytes || C.y_bytes)

challenge = SHA256("LedgerLens/zk/v1" || R0_bytes || R1_bytes || B_bytes || context)
```

Where:
- `wallet` is the Stellar wallet address (UTF-8 bytes).
- `t_byte` is the threshold as a single byte (0–100 fits in 1 byte).
- `C.x_bytes`, `C.y_bytes` are the Pedersen commitment point coordinates as
  32-byte big-endian integers.
- `R0`, `R1` are the Sigma-protocol response points for each bit.
- `B` is the bit commitment point.
- All point encodings are `x_bytes || y_bytes` (64 bytes total), big-endian.

## Wire Format

Proofs are serialised into a fixed-length byte array for on-chain submission:

```
offset  size  field
0       1     version (must be 1)
1       32    score_commit_x (big-endian)
33      32    score_commit_y (big-endian)
65      192   bit record 0  (commit_x, commit_y, c0, c1, s0, s1; 32B each)
257     192   bit record 1
449     192   bit record 2
...           (7 records total, 1345 bytes overall)
```

Each scalar/field element is encoded as 32 big-endian bytes.

## Verification Algorithm

1. Parse the proof structure (validate version, record count, field sizes).
2. Reconstruct the Fiat-Shamir challenge for each bit commitment:
   - Compute `R0 = s0 * H - c0 * B_i`
   - Compute `R1 = s1 * H - c1 * (B_i - G)`
   - Challenge `c = SHA256("LedgerLens/zk/v1" || R0 || R1 || B_i || context)`
   - Verify `c == c0 + c1 mod q`
3. Verify the aggregate sum:
   - Compute `C' = Σ (2^i * B_i)`
   - Compute `C_minus_tG = C - t * G`
   - Verify `C' == C_minus_tG`

If all checks pass, the proof is valid: the committed score is >= threshold.

## Cross-Implementation Testing

The shared test fixture `tests/fixtures/zk_proof_vectors.json` contains proofs
generated by the Python prover. Both the Python and Rust test suites load this
fixture and verify every proof. A change that silently diverges the two
implementations will fail CI in both places.

## Security Considerations

- **Verifier-only design**: The Rust `verify_threshold_proof` function only
  accepts public proof/commitment bytes. It never accepts a secret blinding
  factor.
- **Timing side-channels**: The Python verifier uses `hmac.compare_digest` for
  constant-time point comparison; the Rust verifier relies on `ark-bn254`'s
  constant-time point equality.
- **Wire format validation**: On-chain verifiers must reject unexpected version
  bytes and incorrect payload lengths to prevent format confusion attacks.
